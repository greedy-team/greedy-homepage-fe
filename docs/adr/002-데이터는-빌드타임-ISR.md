# ADR 002. 데이터는 빌드 타임에 읽고 ISR로 갱신한다

- 상태: 확정 (2026-07, 팀 디스코드 논의)
- 결정자: 4기 홈페이지 TF

## 맥락

이 사이트의 데이터는 대부분 기수 단위로 바뀌는 아카이브예요. 백엔드 팀 논의로 도메인별 데이터 출처가 갈렸어요.

- **스터디 커리큘럼**: 기수마다 개발 멤버가 통으로 고치는 문서라 하드코딩(`shared` 또는 스터디 슬라이스)
- **프로젝트, 멤버, 활동**: 데이터 양과 연관관계가 있고, 나중에 GitHub OAuth 로그인이 붙으면 사용자가 직접 쓰는 경로(프로필 셀프 편집, 활동 업로드)가 생겨서 백엔드 API

## 결정

- API를 쓰는 페이지도 **매 요청마다 API를 부르지 않아요.** 빌드 타임에 데이터를 읽어 정적 페이지로 굽고, ISR(revalidate)로 주기 갱신해요.
- 페이지는 `entities`의 api 함수만 불러요. 그 함수가 정적 파일을 읽는지 API를 부르는지는 페이지가 몰라요. API가 준비되기 전에는 정적 초기 데이터로 먼저 공개할 수 있어요.
- 서버 응답(DTO) 타입과 화면이 쓰는 타입은 분리하고, 변환은 `src/api` 안에서 해요.

## 이유

- 방문자는 항상 빠른 정적 페이지를 받아요.
- 다시 굽기가 실패하면 마지막 성공본을 계속 보여줘요. **서버가 내려가도 사이트는 살아 있어요.** 방학 중 서버 관리가 뜸해져도 안전해요.
- 아카이브 데이터에 몇 분의 신선도 차이는 의미가 없어요. 나중에 셀프 편집이 생기면 저장 시 revalidate 웹훅으로 즉시 반영하면 돼요.

## 결과

- 공개 일정이 백엔드 완성 일정에 묶이지 않아요.
- 백엔드에 상시 고가용성 부담이 없어요.
