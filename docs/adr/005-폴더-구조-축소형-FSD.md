# ADR 005. 폴더 구조는 축소형 FSD로 간다

- 상태: 확정 (2026-07)
- 결정자: 4기 홈페이지 TF

## 맥락

운영진이 핵심(MVP)을 만들고, 동아리원이 원하는 기능을 그때그때 붙이는 방향으로 가요. 기여자가 많고 자주 바뀌는 학습 동아리라, 두 가지가 필요해요.

- **기능 격리**: 한 사람이 붙인 기능이 다른 기능을 건드리지 않아야, 안전하게 붙였다 뗐다 할 수 있어요.
- **파급이 보이게**: 공용 기반(디자인 시스템, 도메인)은 여러 화면이 함께 써서, 바뀌면 영향이 넓어요. 어디를 바꾸면 어디까지 번지는지 구조에 드러나야 해요.

타입별로 묶는 구조(`components`, `services`…)는 커지면 한 기능의 UI·데이터·로직이 여러 폴더로 흩어지고, 서로 마음대로 import하면서 얽혀요. 얽힌 의존은 대규모에서 코드를 못 지우고 못 고치게 만들어요.

## 결정

Feature-Sliced Design(FSD)에서 실익 있는 개념만 취한 **축소형**으로 가요. 레이어는 다섯이고, 전부 `src/` 아래 있어요.

| 레이어 | 역할 |
| --- | --- |
| `src/app` | Next 라우팅 + 페이지 조립 + 그 페이지 전용 섹션 |
| `src/widgets` | 여러 페이지가 함께 쓰는 조합 블록 (GNB, 푸터, 모바일 메뉴, 하단 CTA) |
| `src/features` | 기능 단위로 격리해 붙이는 곳. 지금은 비어 있어요 |
| `src/entities` | 도메인 - project·member·activity (api·model·ui) |
| `src/shared` | ui(디자인 시스템)·lib·config·styles·api |

**import은 위에서 아래로 한 방향만** 흘러요. 아래 레이어는 위를 몰라요. 그리고 **`features`끼리는 서로 import하지 않아요.** 이 규칙은 `eslint-plugin-boundaries`로 강제해서, 어기면 CI에서 막혀요.

FSD를 그대로 쓰지 않고 덜어낸 것도 있어요.

| FSD 개념 | 우리 선택 | 이유 |
| --- | --- | --- |
| 세로 슬라이싱 · 단방향 의존 | 취함 | 기능 격리와 대규모 유지보수의 핵심 |
| `widgets` 레이어 | 취함 | GNB·푸터 같은 조합 블록이 이미 있어요 |
| 슬라이스 안 `ui`/`model`/`api` 세그먼트 | 취함(가볍게) | 필요한 세그먼트만 만들어요 |
| `pages`(→`views`) 레이어 + 루트 re-export | 뺌 | Next `app/`이 이미 조립 지점이에요. `src/pages`를 또 두면 재-export 이중 파일과 빈 루트 `pages/` 폴더 우회가 생겨요 |
| FSD `app` 레이어(provider·init) | 뺌 | `app/layout.tsx`가 대신해요 |
| 정통 툴링(`steiger`) | 안 씀 | 배치를 우리 방식으로 정하니, 범용 `eslint-plugin-boundaries`로 같은 규칙을 강제해요 |

## 이유

- 대규모·다기여자에서 코드를 지키는 건 폴더 이름이 아니라 **단방향 의존 강제**예요. 그 알맹이만 가져왔어요.
- `features` 격리 덕에 동아리원이 코어를 건드리지 않고 자기 폴더 하나로 기능을 더해요. 잘못된 의존은 사람이 아니라 lint가 잡아요.
- 세리머니를 줄여야 기여 장벽이 낮아져요. 그래서 공식 가이드의 `src/pages`·재-export는 우리 규모(대여섯 페이지)에서 이득보다 부담이라 뺐어요.
- 나중에 사이트가 훨씬 커지거나 정통 FSD가 필요하면, 축소형이 그 부분집합이라 `src/pages` 도입으로 매끄럽게 승격할 수 있어요.

## 결과

**어디에 뭘 두나**

- 버튼·카드 같은 공용 UI → `shared/ui`
- `cn`·포커스 링 → `shared/lib`, 사이트 상수 → `shared/config`, 토큰 → `shared/styles`
- 프로젝트·멤버·활동 데이터와 타입 → `entities/<도메인>/{api,model}`
- GNB·푸터처럼 여러 페이지가 쓰는 블록 → `widgets`
- 한 페이지에서만 쓰는 섹션 → 그 라우트 아래(`app/<route>`)에 colocate
- 동아리원이 새로 붙이는 기능 → `features/<기능>`

기능을 붙이는 구체적 절차는 [CONTRIBUTING.md](../../CONTRIBUTING.md)의 "기능 추가하기"에 있어요.
